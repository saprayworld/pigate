# Instruction: วิธีสร้างเช็คลิสต์ทดสอบ (Interactive Test Checklist) สำหรับ PiGate

> คู่มือมาตรฐานสำหรับ **คนและ AI** ในการสร้าง "รายงานเช็คลิสต์ทดสอบ" แบบ HTML
> interactive + export JSON เพื่อยืนยันงานหลังลงมือทำเสร็จ โดยเฉพาะส่วนที่
> **ทดสอบอัตโนมัติไม่ได้** ต้องรันบนอุปกรณ์จริง เป้าหมายคือให้ทุกงานทดสอบด้วย
> แนวทางเดียวกัน ผลตรวจสอบย้อนได้ และ AI อ่านผลกลับมาตรวจซ้ำได้
>
> คู่กับ `work-planning-instruction.md` (แผนงาน) — แผนกำหนด Definition of Done
> ส่วนเอกสารนี้กำหนดวิธีพิสูจน์ว่า DoD นั้นเป็นจริงบนเครื่องจริง
>
> ตัวอย่างที่ทำตามมาตรฐานนี้: การทดสอบ cookie-only session auth (issue #29) —
> ผู้ทดสอบกรอกค่า `Set-Cookie` จริงจาก DevTools แล้ว export JSON กลับมาให้
> AI ยืนยันว่า `Secure` ทำงานจริงบน HTTPS

---

## 0. หลักการสำคัญ 3 ข้อ (อ่านก่อนเริ่มทุกครั้ง)

1. **เช็คลิสต์ต้องมาจาก DoD + ข้อควรระวังของแผน ไม่ใช่คิดสดตอนทดสอบ**
   ทุกข้อในเช็คลิสต์ควรสืบย้อนไปถึง §6 (Definition of Done) และ §5 (Cautions)
   ของแผนงานได้ ข้อควรระวังแต่ละข้อในแผน = อย่างน้อยหนึ่งข้อทดสอบ เพราะมันคือ
   "จุดที่รู้อยู่แล้วว่าจะพังได้" ถ้าเช็คลิสต์ไม่ครอบคลุม caution แปลว่ายังไม่พิสูจน์
   ว่ากันความเสี่ยงที่ตัวเองเขียนไว้ได้จริง
   **ข้อยกเว้น:** caution ที่ automation พิสูจน์ได้ครบถ้วนแล้ว (`go test`,
   `verify` skill — ดูหลักข้อ 3) ไม่ต้องมีข้อใน HTML แต่ AI ต้องระบุตอนสรุปผล
   ว่า caution ไหนถูกพิสูจน์ด้วย automation ไปแล้ว เพื่อให้ความครอบคลุม
   ยังตรวจย้อนได้

2. **ข้อสำคัญต้องเก็บ "ค่าที่สังเกตได้" ไม่ใช่แค่ ผ่าน/ไม่ผ่าน**
   checkbox อย่างเดียวเชื่อไม่ได้ — คนกดผ่านโดยไม่ได้ดูจริงก็ได้ ข้อที่เป็น
   หัวใจ (ค่าใน DevTools, response body, header, ค่าใน localStorage) ต้องมี
   ช่องให้ผู้ทดสอบ **วางค่าจริงที่เห็น** ลงไป เพื่อให้คนรีวิว/AI อ่านค่านั้น
   แล้วตัดสินเองได้ ไม่ใช่เชื่อ checkbox เช่น แทนที่จะเชื่อว่า "cookie มี Secure"
   ให้เก็บ header จริง `set-cookie: ...; HttpOnly; Secure; SameSite=Strict`
   มาแล้ว AI ยืนยันเองได้

3. **แยก "ทดสอบอัตโนมัติได้" ออกจาก "ต้องเครื่องจริง" ให้ชัด**
   สิ่งที่ automate ได้ AI ต้องทำเองก่อนเสมอ ไม่ยกภาระให้คน:
   `go build/test`, `yarn build/lint`, และ **การขับ UI จริงด้วย `verify` skill**
   (embedded single-binary + Playwright headless ใน mock mode) เช็คลิสต์ HTML
   มีไว้สำหรับ **ส่วนที่เหลือซึ่ง mock/CI ครอบไม่ถึง** — HTTPS จริง, cert,
   พฤติกรรม browser จริง, อุปกรณ์จริง, role/สิทธิ์ปลายทาง อย่าเอาของที่ automate
   ได้อยู่แล้วมายัดเป็นภาระมือของคน

## 1. เมื่อไหร่ต้องทำเช็คลิสต์แบบนี้

ทำเมื่องานมี "ช่องว่าง" ที่ mock/CI พิสูจน์ไม่ได้ และผลลัพธ์กระทบผู้ใช้จริง:

- งานที่พฤติกรรมจริงขึ้นกับ **สภาพแวดล้อมจริง**: HTTPS/TLS, cert, cookie `Secure`,
  D-Bus/Netlink/kernel behavior, Wi-Fi, DHCP/DNS ปลายทาง, power
- งานที่ต้อง **ยืนยันด้วยตาใน browser จริง**: DevTools (Network/Application/
  Console), พฤติกรรม redirect, SSE, การเด้งหน้า, dark/light
- งานที่แตะ **auth/role/สิทธิ์** — ต้องทดสอบด้วยหลาย role บนเครื่องจริง
- งานที่ **migrate เครื่องที่ติดตั้งไปแล้ว** (install.sh ซ้ำ, key เก่าค้าง)

งานเล็ก/แก้ style จุดเดียว ที่ `verify` skill หรือ unit test ครอบได้หมด **ไม่ต้อง**
ทำเช็คลิสต์ HTML — ใช้วิจารณญาณ

**ที่เก็บไฟล์ report:** วางไฟล์ `.html` ที่ **root ของ repo** (หรือโฟลเดอร์ชั่วคราว)
เพื่อให้ผู้ใช้เปิดง่าย — **เป็น test artifact ไม่ commit เข้า repo** (ยกเว้น
ผู้ใช้ขอให้เก็บ) ไฟล์ JSON ที่ผู้ใช้ export กลับมาก็เป็น artifact เช่นกัน

## 2. ที่มาของข้อทดสอบ (map แผน → เช็คลิสต์)

ไล่จากเอกสารแผนงานของงานนั้น แปลงเป็นข้อทดสอบ:

| แหล่งในแผน | แปลงเป็นข้อทดสอบอย่างไร |
|---|---|
| §6 Definition of Done ทุก checkbox | ข้อไหน automate แล้ว (build/test/lint/`verify`) → AI ทำเอง ไม่ต้องใส่ใน HTML; ข้อที่ต้องตาคนดูบนเครื่องจริง → เป็นข้อในเช็คลิสต์ |
| §5 Cautions แต่ละข้อ | หนึ่ง caution = หนึ่งข้อทดสอบ "พิสูจน์ว่ากันได้จริง" (เช่น caution เรื่อง cookie ถูกทิ้งถ้าไม่มี Secure → ข้อ "cookie มี Secure จริงบน HTTPS") |
| §0 เป้าหมาย (พฤติกรรมผู้ใช้) | เป็นข้อ happy-path หลัก (login → ใช้งาน → logout ฯลฯ) |
| §4 API + role | เป็นข้อทดสอบสิทธิ์ (super_admin ทำได้ / read-only โดนบล็อก 403) |
| กับดักที่เจอตอนสำรวจโค้ด | เป็นข้อ edge-case หรือข้อในหมวด "จุดเสี่ยงที่สุด" |

**ระบุค่าที่ต้อง capture:** ข้อไหนที่ AI จะตรวจซ้ำได้จากค่าจริง ให้ทำเป็นข้อแบบ
มีช่อง observed พร้อม placeholder บอกว่าให้วางอะไร (เช่น "วางค่า Set-Cookie header")

## 3. โครงสร้าง section ของเช็คลิสต์ (เรียงตาม flow การทดสอบ)

จัดกลุ่มข้อตามลำดับที่คนจะทดสอบจริง อย่างน้อยควรมี:

0. **เตรียมก่อนทดสอบ** — build/deploy/setcap/สิทธิ์ ให้พร้อมบนเครื่องจริง
1. **Happy path หลัก** — flow ที่ผู้ใช้ใช้จริงตั้งแต่ต้นจนจบ
2. **ข้อสังเกตชี้ขาด (หัวใจของงาน)** — ข้อที่มีช่อง observed ให้วางค่าจริง
   (DevTools, header, storage) — นี่คือส่วนที่ทำให้ผลตรวจสอบย้อนได้
3. **flow ครบวงจร / หลายหน้า / หลาย role**
4. **Edge cases** — จาก cautions และกับดักที่เจอ
5. **จุดเสี่ยงที่สุด — ยืนยันก่อนเพื่อน** — 2-3 ข้อที่ถ้าพลาดจะพังทั้งงาน
   (แยกให้เด่น เช่นทำเป็นแถบสีเตือน) เพื่อให้ผู้ทดสอบโฟกัสก่อน

แต่ละข้อประกอบด้วย: **label ชัดเจน** (บอกว่าดูที่ไหน คาดหวังอะไร) +
สถานะ ผ่าน/ไม่ผ่าน/ไม่เกี่ยว + ช่องหมายเหตุ + (เฉพาะข้อสำคัญ) ช่อง observed

## 4. ข้อกำหนดไฟล์ HTML report

- **Self-contained ไฟล์เดียว** — inline CSS/JS ทั้งหมด **ห้ามพึ่ง external
  (CDN/font/script)** เพราะต้องเปิดบนเครื่องที่อาจไม่มีเน็ต/ปิด origin
- **เป็นไฟล์ local ห้ามใช้ Artifact ของ claude.ai** — เพราะปุ่ม download/
  export JSON ต้องใช้ Blob + `URL.createObjectURL` ซึ่ง CSP ของ Artifact บล็อก
  ไฟล์ local เปิดตรง ๆ ทำงานได้ทุกฟีเจอร์
- **Interactive:** แต่ละข้อเลือกสถานะได้ (ผ่าน/ไม่ผ่าน/ไม่เกี่ยว), มีช่อง
  หมายเหตุ, ข้อสำคัญมีช่อง observed
- **Metadata header:** ผู้ทดสอบ, วันที่, อุปกรณ์, URL/IP, build/commit, browser
- **Autosave ลง localStorage** — ปิด/เปิด/refresh แล้วข้อมูลไม่หาย ทดสอบข้ามวันได้
  (state ผูกกับ browser + ตำแหน่งไฟล์ตาม `file://` origin — อย่าเปลี่ยน browser
  หรือย้าย/เปลี่ยนชื่อไฟล์ระหว่างทดสอบ ไม่งั้นข้อมูลที่กรอกไว้หาย)
- **แถบ progress** นับ ผ่าน/ไม่ผ่าน/ไม่เกี่ยว เรียลไทม์
- **ปุ่ม Export JSON** (ดาวน์โหลดไฟล์) **+ Copy JSON** (คลิปบอร์ด) **+ ล้างข้อมูล**
- รองรับ dark/light ด้วย `prefers-color-scheme`

**Schema ของ JSON ที่ export (ตายตัว — เพื่อให้ AI อ่านตรวจซ้ำได้):**

```json
{
  "report": "<topic-slug>",
  "branch": "<branch>",
  "generatedAt": "<ISO timestamp>",
  "meta": { "tester": "", "date": "", "device": "", "url": "", "build": "", "browser": "", "overall": "" },
  "overall": "<ค่าเดียวกับ meta.overall — ใส่ซ้ำที่ top-level เพื่อความสะดวก>",
  "summary": { "total": 0, "pass": 0, "fail": 0, "na": 0, "untested": 0 },
  "items": [
    { "id": "<stable-id>", "section": "<n> <title>", "label": "<plain text>",
      "status": "pass|fail|na|untested", "observed": "<ค่าที่ผู้ทดสอบวาง>", "notes": "" }
  ]
}
```

`id` ของแต่ละข้อต้อง **คงที่และสื่อความหมาย** (เช่น `set_cookie`, `ls_no_session`)
เพื่อให้อ้างถึงได้เวลารีวิว

หมายเหตุ: `meta` จะมีเฉพาะ field ที่ผู้ทดสอบพิมพ์จริง — field ที่ไม่ได้แตะจะ
**หายไปจาก JSON** ไม่ใช่ string ว่าง เวลารีวิวถ้าไม่เจอ `build`/`date` ให้ถือว่า
"ไม่ได้บันทึก" แล้วทวงถามถ้าจำเป็น (เทมเพลตจะเตือนผู้ทดสอบตอน export
ถ้าสองช่องนี้ยังว่าง)

## 5. Template พร้อมใช้

ใช้ **`docs/ref/instruction/test-checklist-template.html`** เป็นจุดตั้งต้น —
แก้แค่ 2 จุดบนสุดของ `<script>`:

1. `META` (ชื่อ report, branch, title ที่โชว์)
2. `SECTIONS` (array ของหมวด + ข้อทดสอบ) — แต่ละ item รูปแบบ:
   ```js
   { id: "set_cookie", label: "Set-Cookie มี HttpOnly + Secure + SameSite=Strict",
     type: "value",            // ใส่ "value" ถ้าต้องเก็บค่าที่สังเกต; ตัดออกถ้าแค่ pass/fail
     hint: "บน HTTPS จริงต้องมี Secure",
     vLabel: "วางค่า Set-Cookie header", vPlaceholder: "pigate_session=...; HttpOnly; Secure" }
   ```
   หมวดที่เป็น "จุดเสี่ยงที่สุด" ใส่ `risk: true` ที่ระดับ section เพื่อให้ขึ้นแถบเตือน

ส่วน render / autosave / export ในเทมเพลตทำงานครบแล้ว ไม่ต้องแก้ กลไกสำคัญ:
config-driven (สร้าง DOM จาก `SECTIONS`), เก็บ state ใน `localStorage` ด้วย key
เฉพาะ report, และ `buildReport()` ประกอบ JSON ตาม schema ข้างบน

> **สิ่งที่ห้ามทำในเทมเพลต:** อย่าใส่ `<script src>`/`<link href>` ภายนอก,
> อย่าเปลี่ยน export ให้ยิงขึ้น network, อย่าลบ autosave (ผู้ทดสอบเสียข้อมูลกลางคัน)

## 6. วงจรการใช้งาน (ทั้งคน + AI)

1. **AI สร้าง report** จากแผน (§2-§3) → คัดลอกเทมเพลต แก้ `META`/`SECTIONS`
   วางไฟล์ที่ root repo → ส่งให้ผู้ใช้ด้วย `SendUserFile` (display render)
2. **AI ทดสอบส่วนที่ automate ได้ก่อน** — `go test`, `yarn lint`, `verify` skill
   (mock) → บอกผลให้ผู้ใช้รู้ว่าเหลือเฉพาะข้อบนเครื่องจริงให้เขาทำ
3. **ผู้ใช้ deploy + ทดสอบบนเครื่องจริง** กรอกสถานะ + วางค่า observed → Export JSON
4. **ผู้ใช้แปะ JSON กลับ** → **AI อ่าน `items[].observed` ทีละข้อ ตรวจซ้ำ**
   ว่าค่าจริงตรงกับที่ควรเป็น (ไม่ใช่แค่ดู `status`) เช่น ยืนยันว่า header ที่วางมา
   มี `Secure` จริง, response body ไม่มี `token` จริง, localStorage ไม่มี key ต้องห้าม
5. **AI สรุปผล** — ผ่านครบไหม / ข้อไหนต้องแก้ / ข้อ N/A มีเหตุผลรับได้ไหม →
   ถ้าผ่าน DoD ครบ ค่อยดำเนินการ commit/PR ต่อ

จุดคุ้มค่าที่สุดของแนวทางนี้คือ **ช่อง observed** — มันเปลี่ยน "เชื่อว่าคนทดสอบแล้ว"
เป็น "ตรวจค่าที่คนทดสอบเห็นได้จริง" ทำให้ AI ช่วย verify ปลายทางได้แม้ไม่ได้อยู่หน้าเครื่อง

## 7. ข้อกำหนดการเขียน (Style)

- **ภาษา:** label/hint ในเช็คลิสต์เป็นไทยปนศัพท์เทคนิคอังกฤษ (เหมือน UI ของโปรเจกต์)
  ชื่อไฟล์/​id เป็นอังกฤษ
- **label ต้องบอก "ดูที่ไหน + คาดหวังอะไร"** ไม่ใช่แค่ชื่อฟีเจอร์ เช่น
  "Network → auth/login → Response: ไม่มี field token" ดีกว่า "เช็ค login"
- **หมวดจุดเสี่ยง** ต้องมีเสมอถ้างานมีทางพังเงียบ ๆ (silent failure) — ระบุ
  อาการที่จะเห็นถ้าพัง เพื่อให้ผู้ทดสอบจับได้
- **จำนวนข้อ:** พอดีกับ DoD + cautions อย่ายัดข้อที่ automate ได้อยู่แล้ว
  โดยทั่วไป ~15-25 ข้อกำลังดี
- **id คงที่:** ตั้งครั้งเดียวแล้วอย่าเปลี่ยน เพราะใช้อ้างอิงตอนรีวิว JSON
- **หลังทดสอบเสร็จ:** ไฟล์ report + JSON เป็น artifact — ถามผู้ใช้ว่าจะเก็บเข้า
  repo, ใส่ `.gitignore`, หรือทิ้ง; อัปเดตแผน/security artifact ว่าข้อไหน verified แล้ว
