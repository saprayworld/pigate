ดีไซน์ระบบจัดการ Routing ของ PiGate

คู่มืออธิบายการจัดหมวดหมู่ระบบนำร่องแพ็กเกจ (Routing), สถาปัตยกรรมระดับตัวเชื่อมประสาน (Service Layer) และระบบตรวจจับเหตุการณ์ผ่านระบบปฏิบัติการ Linux

ประเภทของ Routing (Routing Classification)

การจำแนกประเภทของแต่ละเส้นทางเพื่อนำมาแสดงผลและประมวลผลบนระบบ แบ่งออกเป็น 4 รูปแบบหลัก ดังนี้:

Custom

Custom Static Route

เส้นทางที่ผู้ใช้งานระบุขึ้นเองและไม่มีการเชื่อมโยงกับเกตเวย์ (Gateway == "") สำหรับชี้เครือข่าย Subnet ไปยังอินเตอร์เฟซโดยตรง

💾 จัดเก็บลงในฐานข้อมูล SQLite
Custom Gateway

Custom Gateway Route

เส้นทางที่ผู้ใช้งานกำหนดและระบุ IP ของเกตเวย์ปลายทาง หาก IP นี้ตรงกับ Default Gateway หลัก ระบบจะแปลงไปจัดเก็บเป็นคำเฉพาะ "default" เพื่อเปลี่ยนทิศทางแบบไดนามิก

💾 จัดเก็บลงในฐานข้อมูล SQLite
System

System-defined Route

เส้นทางที่ถูกสร้างขึ้นโดยอัตโนมัติจาก Linux Kernel เมื่อตรวจพบการเปิดใช้งานการ์ดเครือข่าย (เช่น local link/local subnet) ผู้ใช้งานทั่วไปไม่สามารถจัดการได้เป็นค่าเริ่มต้น

🔍 ดึงสดแบบไดนามิกจาก Kernel
Default Gateway

Default Gateway Route

เส้นทางออกสู่เครือข่ายภายนอกทั้งหมดปลายทาง 0.0.0.0/0 ถูกควบคุมจากการตั้งค่า Interfaces หลักโดยตรง

⚠️ จัดการผ่านพอร์ตเชื่อมต่อ (Interfaces)

ระบบจัดการ Routing

การควบคุมจัดการตารางเส้นทางภายในหน่วยประมวลผลถูกออกแบบและแบ่งแยกตามสิทธิ์สถาปัตยกรรมออกเป็น 2 ส่วน:

ความปลอดภัยระดับวิกฤต (Critical System Security) การจัดการ System Route จะต้องเปิดผ่าน CLI Flag เริ่มต้นของตัวแอปพลิเคชันหลังบ้าน และต้องเปิดรับรองความยินยอมยืนยันความปลอดภัยบน UI ก่อนเสมอเพื่อป้องกันความล้มเหลวเครือข่ายหลักขาดการติดต่อ

บริการระดับตัวกลาง (Service Layer)

เลเยอร์การสื่อสารที่ผูกโครงสร้างโค้ดส่วนควบคุมหลักที่ไฟล์ backend/internal/service/routing.go ประกอบด้วย API ย่อย:

User Section (การจัดการทั่วไป)

InitApplyConfig() ทำหน้าที่กู้คืนข้อมูลเส้นทางทั้งหมดที่ผู้ใช้เปิดไว้มาเขียนลง Kernel ตอนเริ่มโปรแกรม โดยต้องทำการตรวจสอบความมีอยู่จริงของการ์ดเชื่อมต่อใน Kernel ปัจจุบันก่อนสั่งเขียนจริงเสมอ
GetDatabaseRoutes() เรียกดูรายการเส้นทางทั้งหมดที่ถูกสร้างจากผู้ใช้และมีข้อมูลจัดเก็บอยู่ในตาราง SQLite
GetRouting() ประมวลผลดึงค่าเส้นทางจริงใน Kernel แล้วนำค่าจาก SQLite มาทับสถานะ/เกตเวย์แบบเรียลไทม์ พร้อมตรวจเช็คความสัมพันธ์เป็นประเภท defaultgateway หรือ customgateway
ApplyConfigRoute() ทำหน้าที่บันทึก/อัปเดตเส้นทาง (ทั้งสร้างใหม่และแก้ไข) ลงในฐานข้อมูลของระบบ พร้อมสั่งใช้งานเขียนทับระดับ Kernel เพื่ออัปเดตผลทันที
RemoveConfigRoute() ลบเส้นทางที่ถูกระบุออกจากฐานข้อมูล SQLite และส่งคำสั่งลบเส้นทางออกจาก Kernel โดยอัตโนมัติ

System Section (การจัดการระบบปฏิบัติการ)

AddSystemRouteDirectly() ข้ามฐานข้อมูลและส่งคำสั่งระดับต่ำสั่งเขียนเส้นทางระดับเคอร์เนลลงเครื่องโดยตรง
EditSystemRouteDirectly() สั่งแก้ไขหรือทดสอบ Metric ของเส้นทางใน Linux Kernel โดยตรง
RemoveSystemRouteDirectly() สั่งลบเส้นทางระบบเครือข่ายออกจากเคอร์เนลของ Linux ทันที

Common Section (ส่วนแกนกลางหลัก)

GetKernelRoutes() ทำหน้าที่อ่านข้อมูลตารางเส้นทาง (Routing Table) สดๆ โดยตรงจาก Linux kernel เพื่อความถูกต้อง ใช้เป็นข้อมูลอ้างอิงหลักในระบบ

ระบบเฝ้าระวังการเปลี่ยนแปลง (Monitor Section)

ควบคุมและตรวจจับเหตุการณ์ผ่านโครงข่ายเน็ตเวิร์กของระบบปฏิบัติการ [netlink_monitor.go](file:///home/sapray/Dev/pigate/backend/internal/service/netlink_monitor.go)

การดักจับของ Netlink Monitor

ระบบจะเฝ้าดักฟังข้อมูล Link (สถานะพอร์ตการ์ดเชื่อมต่อ), Address (ข้อมูลไอพี) และ Route (เส้นทางเน็ตเวิร์ก) ของเครื่อง Linux หากมีเหตุการณ์เปลี่ยนหรือสูญหายไป แอปพลิเคชันจะรันการจัดสมานตาราง (Reconcile) กู้คืนค่าคอนฟิกจากฐานข้อมูลให้อยู่ในสถานะเดิมแบบเรียลไทม์

💡 สถานการณ์จำลอง 1: อินเตอร์เฟซพร้อมแต่ไอพีล่าช้า (เช่น Wi-Fi Connect)

ในตอนแรกการ์ดจะ Link Up ทันที แต่จะยังไม่ได้รับ IP (รอ DHCP เจรจา) ซึ่งส่งผลให้ Static Route ยังใส่ลงเคอร์เนลไม่ได้ในจังหวะแรก Netlink Monitor จะตรวจพบความคืบหน้าเมื่อการ์ดนั้นได้รับไอพีสำเร็จ (Address Event) แล้วจึงสั่งรัน Apply Route เพื่อปรับปรุงเขียนเส้นทางตามในคอนฟิกูเรชันย้อนหลังให้อย่างสมบูรณ์

💡 สถานการณ์จำลอง 2: อุปกรณ์ต่อพ่วงสูญหายและเชื่อมต่อกลับเข้ามาใหม่

ในกรณีอุปกรณ์เครือข่ายต่อพ่วงผ่าน USB (เช่น USB Ethernet หรือ USB Wi-Fi) ถูกถอดออก ตัวเคอร์เนลจะลบเส้นทางออกอัตโนมัติ เมื่ออุปกรณ์นั้นถูกเสียบกลับคืนและเคอร์เนลสร้าง Link ขึ้นมาอีกรอบ Netlink Monitor จะทำการอ่านค่าจากฐานข้อมูลแล้วนำไปเซ็ตค่ากู้คืนระบบกลับสู่สถานะพร้อมใช้โดยอัตโนมัติ

ข้อมูลเพิ่มเติม (Special Design Notes)

การนำส่งเส้นทางด้วยโปรโตคอล 120 (PiGate Custom Protocol) ทุกๆ เส้นทางที่มีการเขียนและดูแลจากผู้ใช้งานจะถูกฝังค่าโปรโตคอลของเคอร์เนลเป็นหมายเลข 120 เพื่อให้ตัวประมวลผลรู้และสั่งลบเส้นทางเก่าทิ้งจากตารางระบบปฏิบัติการได้อย่างถูกต้องเมื่อมีการยกเลิกคอนฟิกูเรชันในโปรแกรม
การก้าวผ่านข้อจำกัดระบบ (Direct System Manipulation) หากมีการประเมินผลผ่าน CLI Flag ในขั้นตอนเริ่มต้นรันโปรแกรมระบบจะยินยอมให้ผู้ใช้งานสามารถสั่งแอด เปลี่ยนค่าโปรโตคอลระดับ Kernel เป็น static, kernel, หรือ boot ได้โดยตรงผ่านคำสั่ง Netlink API
