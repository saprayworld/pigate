Firewall Policy

บันทึกการทำงานของระบบ Firewall Policy ใน PiGate ตั้งแต่ข้อมูลโครงสร้างฐานข้อมูล การลากจัดลำดับความสำคัญ (Drag & Drop Priority) จนถึง Kernel nftables

● Backend Go 1.22+ SQLite nftables อัปเดต 2026-06-15
1
Data Model
backend/internal/model/types.go
Go model/types.go
type PolicyRule struct { ID string `json:"id"` // รหัสเฉพาะ (UUID) Name string `json:"name"` // ชื่อกฎความปลอดภัย InInterface string `json:"inInterface"` // การ์ดขาเข้า เช่น "eth0", "wlan0" หรือ "" (Any) OutInterface string `json:"outInterface"` // การ์ดขาออก เช่น "eth0", "wlan0" หรือ "" (Any) Source []string `json:"source"` // รายชื่อ Address Objects ที่อ้างอิง Destination []string `json:"destination"` // รายชื่อ Address Objects ปลายทางที่อ้างอิง Service []string `json:"service"` // รายชื่อ Service Objects (พอร์ต/โปรโตคอล) ที่อ้างอิง Action string `json:"action"` // "ACCEPT" | "DROP" Log bool `json:"log"` // เปิด/ปิดการเก็บบันทึก Log (true/false) Status bool `json:"status"` // true = เปิดใช้งานกฎ, false = ปิดใช้งานกฎ Priority int `json:"-"` // ลำดับความสำคัญการตรวจสอบ (ใช้ใน DB เท่านั้น) }

Database Schema
backend/internal/db/connection.go

ตารางหลัก firewall_policies และตารางเชื่อมโยงแบบ Many-to-Many สำหรับ Address และ Service Objects:

SQL connection.go
CREATE TABLE IF NOT EXISTS firewall_policies ( id TEXT PRIMARY KEY, name TEXT NOT NULL, in_interface TEXT NOT NULL, out_interface TEXT NOT NULL, action TEXT NOT NULL CHECK(action IN ('ACCEPT', 'DROP')), log INTEGER DEFAULT 0 CHECK(log IN (0, 1)), status INTEGER DEFAULT 1 CHECK(status IN (0, 1)), priority INTEGER NOT NULL ); CREATE TABLE IF NOT EXISTS policy_addresses ( policy_id TEXT NOT NULL, address_id TEXT NOT NULL, association_type TEXT NOT NULL CHECK(association_type IN ('SOURCE', 'DESTINATION')), PRIMARY KEY (policy_id, address_id, association_type), FOREIGN KEY (policy_id) REFERENCES firewall_policies(id) ON DELETE CASCADE, FOREIGN KEY (address_id) REFERENCES address_objects(id) ON DELETE RESTRICT ); CREATE TABLE IF NOT EXISTS policy_services ( policy_id TEXT NOT NULL, service_id TEXT NOT NULL, PRIMARY KEY (policy_id, service_id), FOREIGN KEY (policy_id) REFERENCES firewall_policies(id) ON DELETE CASCADE, FOREIGN KEY (service_id) REFERENCES service_objects(id) ON DELETE RESTRICT );
🛡️
การควบคุม Referential Integrity ด้วย ON DELETE RESTRICT
หาก Address Object หรือ Service Object ถูกอ้างอิงโดยกฎข้อใดข้อหนึ่ง ฐานข้อมูล SQLite จะไม่อนุญาตให้ลบวัตถุดังกล่าวทันทีเพื่อป้องกันปัญหากฎค้าง (orphan rules) ในระดับ kernel
3
Rule Evaluation & Reordering Flow
backend/internal/db/repository.go → SaveAllPolicies()

PiGate ประมวลผลกฎจากบนลงล่าง (First-Match Wins) การปรับตำแหน่งด้วยการ Drag & Drop บน Frontend จะบันทึกลำดับผ่าน REST API:

STEP 1
Drag & Drop UI
สลับแถวตารางใน React (@dnd-kit)
STEP 2
PUT /api/policies/reorder
ส่งอาเรย์ของกฎเรียงตามลำดับใหม่
STEP 3
SQLite DB Transaction
อัปเดต priority 1..N ใน DB
STEP 4
POST /api/policies/apply
แปลงและโหลดกฎลง nftables kernel จริง
⚡
First-Match Wins
เมื่อแพ็กเก็ตวิ่งผ่าน หากตรงเงื่อนไขของกฎข้อใด ระบบจะทำการกระทำ (ACCEPT/DROP) และหยุดมองหากฎข้อถัดไปทันที
🛑
Implicit Deny Fallback
หากแพ็กเก็ตตรวจสอบทุกกฎแล้วไม่ตรงกับข้อใดเลย เคอร์เนลจะทำการ DROP แพ็กเก็ตนั้นโดยอัตโนมัติ (Default Chain Policy)

REST API Endpoints
backend/internal/api/router.go · handlers.go
MethodPathHandlerหน้าที่
GET /api/policies HandleGetPolicies ดึงรายชื่อกฎทั้งหมด เรียงตาม priority ASC
POST /api/policies HandleCreatePolicy เพิ่มกฎใหม่ (คำนวณ priority ล่างสุดให้อัตโนมัติ)
PUT /api/policies/{id} HandleUpdatePolicy แก้ไขกฎ (รวมถึงอัปเดตความสัมพันธ์ Many-to-Many ใหม่)
DELETE /api/policies/{id} HandleDeletePolicy ลบกฎและลบความสัมพันธ์ในตารางลูก
PUT /api/policies/reorder HandleReorderPolicies บันทึกลำดับ Priority หลังการลากสลับแถว
POST /api/policies/{id}/toggle-log HandleTogglePolicyLog เปิด/ปิดการเก็บ Log ของกฎข้อนั้น (toggle บิต log)
POST /api/policies/{id}/toggle-status HandleTogglePolicyStatus เปิด/ปิดการทำงานของกฎ (active ↔ inactive)
POST /api/policies/apply HandleApplyPolicies คอมไพล์และนำคำสั่งทั้งหมดโหลดเข้า nftables kernel

Mock Mode vs Production Mode
backend/internal/kernel/mock.go
mockModeพฤติกรรมของ ApplyRules
true เพียงทำการพิมพ์แสดงโครงสร้าง nftables command ที่ประมวลผลได้ลงสู่ backend standard logs เท่านั้น ไม่มีผลกับ OS จริง
false นำกฎไฟร์วอลล์ทั้งหมดที่เปิดสถานะ (Status = true) แปลงเป็น nftables instructions แล้วเขียนลง netlink socket โดยตรง
6
ข้อควรระวัง
⚠️
การอ้างอิงวัตถุที่โดนล็อก
ไม่อนุญาตให้แก้ไขหรือลบวัตถุระบบ (System Predefined Objects เช่น ALL หรือ ICMP) ผ่านการใช้งานปกติ เนื่องจากระบบย่อยอื่นๆ ใช้ระบุการทำงานพื้นฐาน
⚠️
ตารางหลุดเฟรมบนสมาร์ทโฟน
หน้าแสดงกฎไฟร์วอลล์มีจำนวนคอลัมน์และข้อมูลประเภท Multi-select Combobox ค่อนข้างกว้าง จึงมีคำแนะนำให้ผู้ใช้ใช้งานในหน้าจอขนาดกว้างหรือแท็บเล็ตเป็นอย่างน้อย หรือตรวจสอบว่าตารางครอบด้วยสไตล์ overflow-x-auto w-full
💡
การเชื่อมโยง IP address จริง
เมื่อมีการแก้ไขชื่อหรือเพิ่ม/ลด สมาชิกใน Address Object หรือ Service Object ระบบหลังบ้านจะนำไปแมปร่วมกับการดึงข้อมูล SQL ในตารางเชื่อมโยงทันที เพื่ออัปเดตกฎ nftables ให้เป็นปัจจุบันทุกครั้งที่กด Apply
