Static Route

บันทึกการทำงานของระบบ Static Route (เส้นทางเครือข่ายแบบคงที่) การซิงค์สถานะจากเคอร์เนลผ่าน /proc/net/route และการประยุกต์ใช้กับระบบปฏิบัติการ

● Backend Go 1.22+ SQLite Routing Table อัปเดต 2026-06-16
1
Data Model
backend/internal/model/types.go
Go model/types.go
type StaticRoute struct { ID string `json:"id"` // รหัสอ้างอิงเส้นทาง (หรือ canonical route-sys-... สำหรับ kernel-only) Destination string `json:"destination"` // เครือข่ายปลายทาง เช่น "192.168.10.0/24" Gateway string `json:"gateway"` // IP Gateway (ว่างหากเชื่อมต่อแบบ Direct/system) Interface string `json:"interface"` // อินเตอร์เฟสขาออก เช่น "eth0", "wlan0" Metric int `json:"metric"` // ลำดับความสำคัญการตัดสินใจ (sync จาก kernel) Description string `json:"description"` // คำอธิบายประกอบ Status bool `json:"status"` // true = เปิดใช้งาน (เก็บใน DB, ไม่ถูกทับด้วย kernel) Type string `json:"type"` // "system" | "defaultgateway" | "custom" Scope string `json:"scope"` // ขอบเขตเส้นทาง เช่น "global", "link", "host", "site" Src string `json:"src"` // preferred source IP address Proto string `json:"proto"` // โปรโตคอลเส้นทาง เช่น "kernel", "boot", "static", "120" }

OS Sync Flow (Kernel Route Synchronization)
backend/internal/db/repository.go → SyncRoutesFromOS()

การซิงค์ข้อมูลเส้นทางเครือข่ายจากระบบเคอร์เนลมาปรับปรุงลงฐานข้อมูลจะทำงาน เพียงครั้งเดียวตอนเริ่มการทำงานของแอปพลิเคชัน (Startup Sync) เพื่อป้องกันการลูปเขียนทับซ้ำซ้อน โดยมีลำดับขั้นตอนดังนี้:

STEP 1
อ่านตารางเส้นทาง OS
อ่านข้อมูลและสแกนพอร์ต/เส้นทางในเคอร์เนลจริง
STEP 2
เปรียบเทียบ Custom Routes
ตรวจสอบ Custom Route ใน DB กับ OS และอัปเดตสถานะ Active/Inactive
STEP 3
คำนวณ Canonical ID
ใช้ ID เฉพาะเจาะจง route-sys-[dest]-[gw]-[iface] เพื่อไม่ให้เกิด PRIMARY KEY ซ้ำ
STEP 4
ล้างและบันทึก System Routes
ล้างข้อมูลระบบตัวเดิมและบันทึกข้อมูลตารางใหม่เข้า SQLite

Helper Functions
backend/internal/db/repository.go

parseHexIP(hexStr string) → (ip string, err error)

แปลง Hex Little-endian จาก route table ของ Linux เคอร์เนลให้อยู่ในรูป IPv4 Address ปกติ:

little-endian conversion examplerepository.go
ตัวอย่าง hex string: "0022A8C0" 1. แบ่งเป็นคู่: "00", "22", "A8", "C0" 2. กลับลำดับ byte (Little-endian to Big-endian): "C0", "A8", "22", "00" 3. แปลงเลขฐาน 16 เป็นฐาน 10: C0 → 192 A8 → 168 22 → 34 00 → 0 4. คืนค่าผลลัพธ์: "192.168.34.0"

REST API Endpoints
backend/internal/api/router.go · handlers.go
MethodPathHandlerหน้าที่
GET /api/routes HandleGetRoutes ดึงรายชื่อเส้นทางทั้งหมด (รวม kernel routes ที่ไม่อยู่ใน DB)
GET /api/routes/config HandleGetRoutesConfig ดึง flags การตั้งค่า: allowEditSystemRoutes, prioritizeKernelRoutes
POST /api/routes HandleCreateRoute สร้างเส้นทางใหม่ (type = 'custom')
PUT /api/routes/{id} HandleUpdateRoute แก้ไขเส้นทาง (ล็อกสำหรับ system เว้นแต่มี flag)
DELETE /api/routes/{id} HandleDeleteRoute ลบเส้นทาง (ล็อกสำหรับ system เว้นแต่มี flag)
POST /api/routes/bulk-delete HandleBulkDeleteRoutes ลบหลายเส้นทางพร้อมกัน (สกัดกั้นหากมี system ปนอยู่)
POST /api/routes/{id}/toggle HandleToggleRoute เปิด/ปิด route — defaultgateway/custom Toggle ได้เสมอ; system ต้องมี flag
POST /api/routes/apply HandleApplyRoutes นำการเปลี่ยนแปลงบนหน้าจอไปเขียนทับลงตารางเส้นทาง OS จริง

Config Flags (CLI & Runtime)
backend/cmd/pigate/main.go · GET /api/routes/config
FlagDefaultผลกระทบ
-allow-edit-system-routes false เมื่อ true — ผู้ใช้สามารถแก้ไข ลบ และ Toggle เส้นทาง system ได้
-prioritize-kernel-routes Experimental false เมื่อ true (ไม่เสถียร) — kernel เป็น source of truth สำหรับ metric, type, scope และ proto แต่ status ยังคงมาจาก DB (user intent)
*ข้อควรระวัง: หากเปิดใช้งาน ควรเปิดใช้งาน -allow-edit-system-routes ควบคู่กันไป เนื่องจากหากผู้ใช้สร้าง route แบบ Directly (ไม่มี gateway) เอง ระบบจะมองว่าเส้นทางนั้นเป็นของระบบ (system) และจะไม่สามารถแก้ไขหรือลบผ่านทางหน้าจอหลักได้
💡
ดูค่า flags ปัจจุบันได้ที่ GET /api/routes/config
Frontend ใช้ endpoint นี้เพื่อปรับการแสดงผล (เช่น ซ่อน/แสดงปุ่ม Edit สำหรับ system routes)

Mock Mode
backend/internal/kernel/mock.go
mockModeพฤติกรรมของ ApplyRoutes
true ทำการดึงและปรับปรุงข้อมูลในฐานข้อมูลตามปกติ แต่ไม่มีการเรียกแก้ไข Routing Table ของเครื่องแม่ข่ายจริง
false เรียกใช้ฟังก์ชันเคอร์เนลผ่าน Netlink Socket เพื่อเพิ่ม/ลบเส้นทางในระบบปฏิบัติการ Linux จริง
⚠️
ข้อพึงระวังเกี่ยวกับ prioritizeKernelRoutes (สถานะ Experimental)
ฟังก์ชัน prioritizeKernelRoutes ขณะนี้มีสถานะไม่เสถียร (Experimental) และถูกตั้งค่าเริ่มต้นเป็น false หากเปิดใช้งาน (เป็น true) ควรเปิด -allow-edit-system-routes ควบคู่ไปด้วยเสมอ เนื่องจาก:
  • เมื่อผู้ใช้งานสร้างเส้นทางเครือข่ายแบบเชื่อมตรง (Directly connected route เช่น ไม่มี Gateway) ด้วยตนเอง เคอร์เนลจะมองเส้นทางดังกล่าวเป็น Link-local / System route
  • เมื่อระบบซิงค์ข้อมูลจะปรับประเภท (Type) ในฐานข้อมูลจาก custom ไปเป็น system ส่งผลให้ผู้ใช้ไม่สามารถเข้าไปลบหรือแก้ไขเส้นทางนั้นผ่านหน้าจอหลักได้ (ถูกล็อก) เว้นแต่จะเปิด flag อนุญาตแก้ไข System route

⚙️
Route Type Priority & Behavior
Typeแก้ไขได้Toggle ได้ลบได้ที่มา
system ❌ ต้องมี flag ❌ ต้องมี flag ❌ ต้องมี flag เส้นทาง directly-connected จากเคอร์เนล
defaultgateway ✅ เสมอ ✅ เสมอ ✅ เสมอ เส้นทาง 0.0.0.0/0 (default route)
custom ✅ เสมอ ✅ เสมอ ✅ เสมอ เส้นทางที่ผู้ใช้สร้างเอง
⚙️
การควบคุมและแก้ไข System Routes (ผ่าน Flag ระบบ)
โดยปกติเส้นทางเครือข่ายที่เป็นแบบ system จะเป็นแบบอ่านอย่างเดียวเพื่อความปลอดภัยในการป้องกันระบบหลุดการเชื่อมต่อ อย่างไรก็ตาม ผู้ใช้สามารถกำหนดให้ระบบอนุญาตให้แก้ไขหรือลบเส้นทางระบบได้ โดยเริ่มการทำงาน backend ด้วย flag -allow-edit-system-routes ซึ่งระบบจะทำการปลดล็อกให้แก้ไข เพิ่ม และลบ พร้อมทั้งนำไปปรับใช้ในเคอร์เนลจริง
⚠️
รูปแบบ CIDR เครือข่ายปลายทาง
อินพุตช่องปลายทาง (Destination) ต้องกรอกในรูปแบบ CIDR เครือข่ายปลายทางที่ถูกต้องเท่านั้น (เช่น 192.168.5.0/24) หากกรอกเป็นเพียง IP เฉยๆ (เช่น 192.168.5.2) หลังบ้านจะแจ้งเตือนความผิดพลาด

Kernel Integration (Production)
backend/internal/kernel/real_routing.go

การปรับปรุงตารางเส้นทางจริงบนระบบปฏิบัติการ Linux ของ Raspberry Pi 5 จะใช้ไลบรารี netlink ของ Go ในการติดต่อสื่อสารโดยตรงแทนการพิมพ์คำสั่งเชลล์ (Shell commands) ซึ่งมีความรวดเร็วและปลอดภัยสูงกว่า:

Go (Reconciliation Logic) real_routing.go
// 1. ดึงตารางเส้นทางปัจจุบันในเคอร์เนล existingRoutes, err := netlink.RouteList(nil, netlink.FAMILY_V4) // 2. ตรวจสอบการจัดการตามเงื่อนไข (Reconciliation) for _, rt := range existingRoutes { // ตรวจสอบโปรโตคอล (ใช้ Protocol 120 เป็นสัญลักษณ์บ่งบอกว่าสร้างโดยแอปพลิเคชันนี้) isManagedRoute := rt.Protocol == 120 if isActiveTarget { // อัปเดต Metric หรือแก้ไขข้อมูลถ้าหากมีการเปลี่ยนแปลง netlink.RouteReplace(&rt) } else { // ลบเส้นทางหากไม่ได้เปิดใช้งาน (Disabled) หรือถูกลบออกไปแล้ว if isManagedRoute || (isSystemRoute && r.allowEditSystemRoutes) { netlink.RouteDel(&rt) } } } // 3. เพิ่มเส้นทางใหม่ที่เปิดใช้งาน for _, route := range activeRoutesToApply { // ค้นหาอินเตอร์เฟสและแปลงที่อยู่ IP (หากขัดข้องจะใช้การ Log Warning และประมวลผลต่อเพื่อไม่ให้บล็อกรายการอื่น) netlinkRoute := &netlink.Route{ LinkIndex: link.Attrs().Index, Dst: dstNet, Gw: gwIP, Priority: route.Metric, Protocol: 120, } netlink.RouteAdd(netlinkRoute) }
